一 · 项目概览

这是一套前后端分离的农产品溯源系统。它管理产品、批次及其生产 / 质检 / 物流履历，允许消费者扫码查看产品的流转过程，并在此基础上叠加了一层区块链式防篡改机制——基于 SHA-256 数据指纹和链式审计日志实现。

系统能做什么

数据概览

统计产品、批次、溯源记录数量，并展示近期趋势图表。

产品管理

增删改查 + 分页搜索；每个产品都携带一份 data_hash 数据指纹。

批次管理

增删改查、按产品筛选；批次同样有指纹，并拥有自己的溯源记录。

溯源详情

公开页面，展示生产、质检、物流记录及二维码入口。

数据指纹

重算并比对哈希，展示全局根哈希与异常项。

审计日志链

对每次 CREATE/UPDATE/DELETE 形成 previous_hash → data_hash 链式日志。

用户管理

管理员维护用户、角色、状态与密码。

统计分析

类别分布 + 溯源趋势，用 ECharts 渲染。

个人中心

昵称/手机号、修改密码、上传头像。

五 · 请求生命周期

每个需要登录的接口调用都走同一条路径。理解这一条，就理解了所有功能。

六 · 登录与会话认证

采用 Spring Security 表单登录 + 服务端会话（JSESSIONID），可选写入持久化的“记住我”Cookie。前端不持有任何令牌——它依赖 Cookie，仅缓存展示信息。

双层守卫

前端路由（beforeEach）是便利性守卫：它调用 GET /api/user/info 确认会话真实存在（这样手改 localStorage 无法伪造登录），再检查 meta.roles。

关键点：真正的权限拦截在服务端 SecurityConfig。路由守卫只是给出友好跳转；即便客户端被篡改，接口仍会返回 401/403。

七 · 产品增删改与指纹生成

最重要的写入路径。每次增/改/删都会：(1) 设定稳定的 id 与时间，(2) 按固定字段顺序计算 data_hash，(3) 追加一条链式审计日志——全部在同一个事务内完成。

指纹公式（Java 与 SQL 必须完全一致）

data_hash = SHA-256( id | name | category | origin | 归一化价格 | createTime )
// 价格去掉末尾的 0：12.50 与 12.5 → "12.5"（指纹相同）
为什么三处必须保持一致：ProductService.computeProductHash、schema.sql 的回填语句、以及 BlockchainSchemaInitializer 计算的是同一个字符串。一旦字段顺序或价格归一化方式跑偏，同一行数据在 Java 与 SQL 中会算出不同哈希，完整性校验就会误报。
防篡改的设计意图：正常编辑都经过服务层，会重算哈希。要模拟篡改，必须直接改数据库的业务字段而不更新 data_hash——此时数据指纹页和审计日志校验都会把它标记为异常。

八 · 数据指纹校验

回答“当前业务数据是否还是它声称的那份？”它对每一行实时重算哈希，与存储的 data_hash 比对；同时生成一个全局根哈希。

数据指纹 vs. 审计日志链 —— 两个不同的问题

校验它回答的问题接口
数据指纹业务行是否被系统外修改过？（当前数据 ≠ 存储哈希）/api/integrity/*
审计日志链日志表本身是否被断链/插入/篡改？/api/blockchain/logs/verify

审计校验接口会同时跑两者：如果链条完整但某产品被直接改库，它依然会提示“日志链完整，但当前业务数据有 N 项指纹异常”。

九 · 审计日志链（区块链模拟）

每次写入都追加一条 blockchain_log，其哈希包含上一条的哈希——就像一个区块引用它的父区块。插入、删除或篡改任意中间一条都会破坏链条。

verify() 逻辑

for 每条日志（按 id 升序）:
   content   = actionType + targetId + operator + timestamp + previousHash + dataAfter
   calcHash  = SHA-256(content)
   if calcHash != log.dataHash        → “日志自身哈希不一致，可能被篡改”（brokenIndex）
   if previousHash != log.previousHash → “上一哈希不连续，日志链已断裂”
   previousHash = log.dataHash
// 链条通过 ⇒ 再对产品 + 批次运行 verifyBusinessData()
return { logChainValid, dataIntegrityValid, invalidItems, message }

十五 · 启动方式

后端

# 需要 JDK 17+、Maven、MySQL 8（:3306）
cd backend
mvn spring-boot:run     # → http://localhost:8080
mvn test                # 运行测试

首次启动会自动创建 agriculture_trace 库、执行 schema.sql，并回填指纹/日志。ddl-auto=none；默认账号 root / 123456（可用 MYSQL_* 环境变量覆盖）。

前端

cd frontend
npm install
npm run dev     # → http://localhost:5173
npm run build   # 生产构建 / 验证

接口地址在 frontend/.env.development 中配置 → VITE_API_BASE_URL=http://localhost:8080。

十六 · 术语表

术语含义
data_hash 数据指纹对一行业务字段做的 SHA-256；用于发现系统外的直接改库。
root hash 根哈希把所有产品指纹拼接后再做一次 SHA-256——一个全局完整性印章。
blockchain_log 审计日志只追加的账本；每行哈希都包含上一行的哈希。
previous_hash 前哈希上一条日志的 data_hash（创世块为 "0"）；把链条串起来。
Result<T>统一接口信封 {code, message, data}；code 401 = 会话过期。
JSESSIONID服务端会话 Cookie；登录的真正凭据（而非 localStorage）。
remember-me 记住我可选的持久化 Cookie（7 天），用于保持登录。
create-with-trace同时创建产品 + 批次 + 溯源记录的单一接口。
