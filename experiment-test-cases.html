农产品溯源系统实验用例

本用例根据当前项目代码编写，覆盖 Spring Boot 后端、Vue 3 前端、MySQL 初始化数据、Session 登录、权限控制、产品/批次管理、溯源、数据指纹和审计日志。

后端：backend，默认端口 8080
前端：frontend，默认端口 5173
数据库：agriculture_trace，MySQL 8+

1. 测试依据

本实验用例基于当前项目的实际页面、接口和数据库脚本编写，主要依据如下：

  • 前端路由：frontend/src/router/index.js。
  • 产品页面：frontend/src/views/ProductList.vue、frontend/src/components/ProductFormDialog.vue。
  • 批次页面：frontend/src/views/BatchList.vue。
  • 溯源页面：frontend/src/views/TraceDetail.vue。
  • 数据指纹页面：frontend/src/views/IntegrityReport.vue。
  • 审计日志页面：frontend/src/views/blockchain/AuditLog.vue。
  • 接口控制器：ProductController、BatchController、TraceController、IntegrityController、BlockchainLogController、UserController、DashboardController。
  • 安全配置：SecurityConfig，使用 Session/Cookie 登录，部分接口按角色授权。
  • 数据库初始化：backend/src/main/resources/schema.sql。
重要规则：产品和批次的 data_hash 不会自然过期。只有正常业务新增/修改时才应重算；直接改数据库业务字段但不改 data_hash 应被检测为异常。

2. 环境与初始数据

2.1 启动顺序

  1. 启动 MySQL。
  2. 启动后端：cd backend，执行 mvn spring-boot:run。
  3. 启动前端：cd frontend，首次执行 npm install，然后执行 npm run dev。
  4. 浏览器访问 http://localhost:5173。

2.3 初始核心数据

  • 产品：prod_1 有机苹果、prod_2 五常大米、prod_3 日照绿茶、prod_4 有机西兰花、prod_5 土鸡蛋。
  • 批次：batch_1 到 batch_23，其中有机苹果、五常大米等产品有多个批次。

  • 产品表和批次表均包含 data_hash；审计日志表为 blockchain_log。

5. 数据指纹与审计日志实验用例

以下篡改用例必须直接操作数据库，不能通过前端编辑。通过前端正常编辑会重新计算合法哈希，这是正常业务流程，不是篡改。
编号优先级场景前置条件操作步骤预期结果
HASH-001 P0 产品单条哈希验证正常 数据库未被篡改。 产品管理页对 prod_1 点击“验证哈希”。 提示产品哈希一致，数据未发现篡改。
HASH-002 P0 批次单条哈希验证正常 数据库未被篡改。 批次管理页对 batch_1 点击“验证”。 提示批次哈希一致，数据未发现篡改。
HASH-003 P0 产品一键验证正常 以 admin 或 inspector 登录。 进入数据指纹页或产品管理页点击一键验证。 返回 valid=true，异常项数量为 0；全局根哈希可复制。
HASH-004 P0 直接篡改产品字段 记录 prod_1 当前名称和价格。 在 MySQL 执行：
UPDATE product
SET price = 99.99
WHERE id = 'prod_1';
 不修改 data_hash。产品验证应提示哈希不一致；数据指纹页异常列表出现 prod_1，展示存储指纹和当前指纹。
HASH-005 P0 直接篡改批次字段 记录 batch_1 当前备注。 在 MySQL 执行：
UPDATE batch
SET remark = '数据库直接篡改备注'
WHERE id = 'batch_1';
 不修改 data_hash。批次验证应提示哈希不一致；批次一键验证异常项包含 batch_1。
HASH-006 P0 篡改后重启后端 已执行 HASH-004 或 HASH-005。 停止并重启后端，再重新点击一键验证。 异常仍然存在。启动流程不得自动重算已有 data_hash 来掩盖篡改。
HASH-007 P0 篡改后无变化保存 已直接篡改 prod_1 或 batch_1。 打开对应编辑弹窗，不修改任何字段，直接点击保存。 后端不重算 data_hash，不新增 UPDATE 审计日志，异常仍可被验证发现。
HASH-008 P1 管理员修复业务数据 已知可信原始值，例如 prod_1 原价 12.50。 通过前端编辑把被篡改字段改回可信值并保存。 保存后系统重新计算哈希并写入 UPDATE 审计日志；再次验证通过。
HASH-009 P1 审计日志列表 以 admin 或 inspector 登录。 进入审计日志页面。 展示 CREATE、UPDATE、DELETE 日志，包含目标类型、目标 ID、操作人、时间、哈希和上一条哈希。
HASH-010 P0 审计日志链验证正常 未直接篡改 blockchain_log。 审计日志页点击验证链条完整性。 返回日志链完整；如果业务数据也未被篡改，则业务数据指纹状态也正常。
HASH-011 P0 篡改审计日志内容 备份当前数据库或准备恢复。 直接修改任意 blockchain_log.data_after 或 previous_hash。 审计日志链验证失败，提示链条不完整或日志哈希异常。
HASH-012 P1 只篡改业务表不破坏日志链 已执行 HASH-004。 审计日志页验证链条完整性。 日志链本身可能仍完整，但联动业务数据指纹检查应提示产品或批次业务数据异常。

7. 验收标准

  • 所有 P0 用例必须通过；若失败，视为核心功能不可交付。
  • P1 用例建议全部通过；若有失败，需要记录影响范围和修复计划。
  • P2 用例用于体验和边界验证，不影响主流程交付，但应在最终答辩或演示前尽量通过。
  • 执行指纹篡改类用例前，建议备份数据库，避免实验数据污染后续演示。
  • 所有通过前端正常新增/编辑的数据，产品/批次哈希验证应一致；所有直接改库但不改 data_hash 的数据，应能被验证发现。
