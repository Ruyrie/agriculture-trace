2区块链能力总览：两套机制，三个产物

本系统并非真正的分布式区块链，而是用关系数据库模拟区块链的核心思想——不可篡改与可验证。它由两套独立又联动的机制构成。

机制 A

数据指纹（Data Fingerprint）

对 product / batch 的关键字段做 SHA-256，存入 data_hash。校验时按同样字段即时重算，与存储值比对——可发现「绕过系统直接改库」的篡改。

产物：每行业务数据的 data_hash + 全局 rootHash

机制 B

审计日志链（Audit Log Chain）

每次 CREATE/UPDATE/DELETE 写一条 blockchain_log，其哈希包含上一条日志的哈希。插入、删除或修改中间任意日志，都会让后续链条断裂。

产物：previous_hash → data_hash 的单向哈希链

两者的分工：「数据指纹」校验当前业务数据是否与存储指纹一致；「审计日志链」校验日志表本身是否连续未被篡改。审计接口 /api/blockchain/logs/verify 在链条通过后会联动调用业务数据指纹校验——即使日志链完整，若有人直接改库也会提示「业务数据指纹异常」。

3数据指纹逻辑

核心代码：ProductService.computeProductHash / BatchService.computeBatchHash / IntegrityController。

3.1 指纹计算规则

用固定顺序的字段，以 | 分隔拼成字符串后做 SHA-256。字段顺序与归一化规则在 Java、schema.sql、启动迁移三处必须严格一致，否则同一条数据会算出不同哈希。

// ProductService.computeProductHash —— 产品指纹 HashUtil.sha256(String.join("|", id, name, category, origin, normalizePrice(price), // 价格去尾零：12.50 与 12.5 视为同值 createTime)); // BatchService.computeBatchHash —— 批次指纹 HashUtil.sha256(String.join("|", id, batchNo, product.id, // 用产品ID而非产品名：产品改名不影响已有批次指纹 productionDate, remark, createTime));
为什么先定 ID 和 createTime 再算哈希？ 这两个字段也是指纹的一部分。create() 中先 setId(nextProductId())、setCreateTime(...)，最后才 setDataHash(...)——若顺序颠倒，保存后的数据将永远无法通过完整性校验。

3.2 篡改检测原理

图 2 · 数据指纹篡改检测 —— 正常业务操作会同步重算指纹（绿色路径不会误报）；只有「直接改库不改 data_hash」才会被识别为异常（红色路径）。

4审计日志链逻辑

核心代码：BlockchainLog 实体、ProductService.recordLog / BatchService.recordLog（写入）、BlockchainLogController.verify（校验）、Ids.logId（单调主键）。

4.1 链式结构

每条日志的 data_hash 由以下内容拼接后 SHA-256：actionType + targetId + operator + timestamp + previousHash + dataAfter。其中 previousHash 取上一条日志的 data_hash，创世块为 "0"。

图 3 · 审计日志链 —— 每块的哈希「咬住」前一块的哈希。改动任何一条历史日志，其 hash 改变，后续所有 previousHash 不再连续，校验立即定位到断裂点。

4.2 写入与校验代码

写入（recordLog，产品/批次共用同一套逻辑）

BlockchainLog log = new BlockchainLog(); log.setId(Ids.logId()); // 时间前缀+自增， // 保证按id排序=写入顺序 log.setActionType(action); log.setTargetId(targetId); log.setOperator(currentOperator()); log.setDataAfter(toJson(after)); log.setTimestamp(now); var last = logRepository.findLastLog(); String prev = last==null ? "0" : last.getDataHash(); log.setPreviousHash(prev); log.setDataHash(HashUtil.sha256( action+targetId+operator+timestamp+prev+dataAfter)); logRepository.save(log);

校验（BlockchainLogController.verify）

String previousHash = "0"; for (log : logs) // 按 id 升序遍历 String content = actionType+targetId+operator +timestamp+previousHash+dataAfter; String calc = HashUtil.sha256(content); if (!calc.equals(log.dataHash)) // 内容被改 || !previousHash.equals(log.previousHash)) // 链断 → 返回 brokenIndex / brokenLogId previousHash = log.dataHash; // 推进到下一块 // 链通过后，联动校验业务数据指纹 verifyBusinessData(); // product+batch 全量比对
为什么按 id 排序而不是 timestamp？ timestamp 是本地墙钟字符串，跨时区/同一秒多条会乱序，导致「验证遍历顺序」≠「写入链尾顺序」，从而误报「上一哈希不连续」。Ids.logId() 用 13 位 UTC 毫秒 + 6 位进程自增 + 13 位随机后缀，字典序严格等于写入顺序。

4.3 校验结果语义

logChainValiddataIntegrityValid含义前端提示
truetrue日志链完整 + 业务数据指纹一致验证结果：正常
truefalse日志连续，但有人直接改库日志链完整，但有 N 项指纹异常
false—日志被插入/删除/篡改断裂位置：第 X 条（日志ID…）

5全局根哈希（RootHash）

核心代码：IntegrityController.calculateRootHash。

把所有产品按 id 升序的 currentHash 用 | 串联后再做一次 SHA-256，得到一个代表「全局数据摘要」的根哈希。任一产品指纹变化都会引起根哈希变化。

// 简化版根哈希（非完整 Merkle Tree，但能模拟全局摘要校验） String joined = rows.stream() .map(r -> r.currentHash) .reduce("", (l,r) -> l.isEmpty()? r : l+"|"+r); rootHash = HashUtil.sha256(joined);
前端「数据指纹」页顶部展示根哈希并支持一键复制（IntegrityReport.vue · copyRootHash），用作整体数据完整性的「快照指纹」。固定按 id 升序保证 rootHash 稳定可复现。

6溯源数据逻辑

核心代码：TraceController、TraceDataService、三类记录实体与仓库、前端 TraceDetail.vue。

6.1 数据归属关系

图 4 · 溯源数据模型 —— 批次是溯源核心业务单元，生产/质检/物流记录均以 batch_id 挂载；访问溯源页时写 TraceRecord 供仪表盘趋势统计。

6.2 关键设计

  • 一次性写入：新增产品的复合接口 /api/product/create-with-trace 在同一事务内创建产品、批次、三类溯源记录，避免「产品有了但溯源不完整」的半成品。
  • 先删后写：编辑批次溯源时 replaceTraceRecords 先删旧记录再整体写入，靠外层事务保证失败回滚。
  • sortOrder 保序：保存前端录入顺序，溯源详情页按真实业务顺序展示。
  • 兼容别名：返回行同时给 item/date/node/time 等旧字段名，避免前端改版断接口。
  • 公开访问：/api/trace/** 在 SecurityConfig 中 permitAll()，支撑二维码无需登录扫码溯源。

7关键流程时序

7.1 新增产品（带溯源）→ 自动产生指纹与日志

图 5 · 新增产品时序 —— 业务数据落库的同时自动生成 data_hash 指纹并追加审计日志链；整个复合操作在一个事务内完成。

7.2 验证链条完整性（审计日志页「验证」按钮）

  1. 前端 AuditLog.vue · verifyChain() → GET /api/blockchain/logs/verify
  2. 后端按 id 升序取全部日志，从 previousHash="0" 开始逐条重算并比对 → 任一断裂返回 brokenIndex/brokenLogId
  3. 链条通过后联动 verifyBusinessData()，对全部产品/批次重算指纹 → 汇总 invalidItems
  4. 前端用 logChainValid 显示链条状态标签，用 valid 显示总体结论，并列出指纹异常明细
