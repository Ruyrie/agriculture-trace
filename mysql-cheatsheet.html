agriculture_trace 数据库

农产品溯源系统 MySQL 常用语句完整速查

这份文件整理了项目中最常用的 MySQL 语句，包括产品、批次、溯源、数据指纹、 审计日志、模拟篡改、恢复数据和统计分析。适合实验演示、答辩讲解和日常排查。

数据库名 agriculture_trace
核心业务表 product / batch
溯源明细表 production / inspection / logistics
防篡改表 blockchain_log

一、连接与基础命令

用于进入 MySQL、切换数据库、查看表结构。

连接 MySQL

CREATE DATABASE IF NOT EXISTS `agriculture_trace`
DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;

USE `agriculture_trace`;

SHOW TABLES;

DESC `product`;

答辩说法： 项目数据库使用 agriculture_trace，字符集使用 utf8mb4，可以正确保存中文产品名、产地、备注和物流节点。

三、产品常用语句

产品表是溯源的入口，也是产品数据指纹的来源。

1. 查询产品列表

按 ID 查看所有产品 
SELECT
  id,
  name,
  category,
  origin,
  price,
  create_time,
  data_hash
FROM `product`
ORDER BY id ASC;

2. 按名称或产地搜索

模糊搜索产品 
SELECT *
FROM `product`
WHERE name LIKE '%苹果%'
   OR origin LIKE '%山东%';

3. 新增产品测试数据

注意： 真实业务新增建议走前端或后端接口，因为服务层会自动生成合法 data_hash 和审计日志。 下面 SQL 更适合数据库实验。
手动插入产品并计算产品指纹 
INSERT INTO `product`
(`id`, `name`, `category`, `origin`, `price`, `create_time`, `data_hash`)
VALUES
(
  'prod_demo',
  '演示苹果',
  '水果',
  '山东烟台',
  12.50,
  '2026-06-16 10:00:00',
  SHA2(CONCAT(
    'prod_demo', '|',
    '演示苹果', '|',
    '水果', '|',
    '山东烟台', '|',
    '12.5', '|',
    '2026-06-16 10:00:00'
  ), 256)
);

4. 修改产品字段

篡改演示： 如果只改业务字段，不同步更新 data_hash，数据指纹页面会显示异常。
模拟直接改库篡改产品产地 
UPDATE `product`
SET origin = '篡改产地'
WHERE id = 'prod_1';
正常修复产品指纹 
UPDATE `product`
SET data_hash = SHA2(CONCAT(
  IFNULL(id, ''), '|',
  IFNULL(name, ''), '|',
  IFNULL(category, ''), '|',
  IFNULL(origin, ''), '|',
  CASE
    WHEN price IS NULL THEN ''
    WHEN price = 0 THEN '0'
    ELSE TRIM(TRAILING '.' FROM TRIM(TRAILING '0' FROM CAST(price AS CHAR)))
  END, '|',
  IFNULL(create_time, '')
), 256)
WHERE id = 'prod_1';

四、批次常用语句

批次是生产、质检、物流三类溯源记录的连接点。

1. 查询批次列表并关联产品

批次加产品名称 
SELECT
  b.id,
  b.batch_no,
  p.name AS product_name,
  p.origin,
  b.production_date,
  b.remark,
  b.create_time,
  b.data_hash
FROM `batch` b
JOIN `product` p ON p.id = b.product_id
ORDER BY b.production_date DESC, b.id ASC;

2. 查询某个产品的所有批次

按产品 ID 查询批次 
SELECT
  b.id,
  b.batch_no,
  b.production_date,
  b.remark,
  b.data_hash
FROM `batch` b
WHERE b.product_id = 'prod_1'
ORDER BY b.create_time DESC;

手动插入批次并计算批次指纹

4. 模拟批次篡改

只改批次备注，不改 data_hash

SET remark = '篡改后的批次备注'
WHERE id = 'batch_1';

UPDATE `batch`
SET data_hash = SHA2(CONCAT(
  IFNULL(id, ''), '|',
  IFNULL(batch_no, ''), '|',
  IFNULL(product_id, ''), '|',
  IFNULL(DATE_FORMAT(production_date, '%Y-%m-%d'), ''), '|',
  IFNULL(remark, ''), '|',
  IFNULL(create_time, '')

六、数据指纹语句

用于理解和演示 data_hash 的计算与校验。

1. 计算产品当前指纹并和存储指纹对比

产品指纹校验 SQL 
SELECT
  id,
  name,
  data_hash AS stored_hash,
  SHA2(CONCAT(
    IFNULL(id, ''), '|',
    IFNULL(name, ''), '|',
    IFNULL(category, ''), '|',
    IFNULL(origin, ''), '|',
    CASE
      WHEN price IS NULL THEN ''
      WHEN price = 0 THEN '0'
      ELSE TRIM(TRAILING '.' FROM TRIM(TRAILING '0' FROM CAST(price AS CHAR)))
    END, '|',
    IFNULL(create_time, '')
  ), 256) AS current_hash,
  CASE
    WHEN data_hash = SHA2(CONCAT(
      IFNULL(id, ''), '|',
      IFNULL(name, ''), '|',
      IFNULL(category, ''), '|',
      IFNULL(origin, ''), '|',
      CASE
        WHEN price IS NULL THEN ''
        WHEN price = 0 THEN '0'
        ELSE TRIM(TRAILING '.' FROM TRIM(TRAILING '0' FROM CAST(price AS CHAR)))
      END, '|',
      IFNULL(create_time, '')
    ), 256)
    THEN '一致'
    ELSE '异常'
  END AS verify_result
FROM `product`
ORDER BY id ASC;

2. 查询产品指纹异常项

只显示产品异常项 
SELECT *
FROM (
  SELECT
    id,
    name,
    origin,
    data_hash AS stored_hash,
    SHA2(CONCAT(
      IFNULL(id, ''), '|',
      IFNULL(name, ''), '|',
      IFNULL(category, ''), '|',
      IFNULL(origin, ''), '|',
      CASE
        WHEN price IS NULL THEN ''
        WHEN price = 0 THEN '0'
        ELSE TRIM(TRAILING '.' FROM TRIM(TRAILING '0' FROM CAST(price AS CHAR)))
      END, '|',
      IFNULL(create_time, '')
    ), 256) AS current_hash
  FROM `product`
) t
WHERE stored_hash <> current_hash;

3. 批次指纹校验

查询批次异常项 
SELECT *
FROM (
  SELECT
    id,
    batch_no,
    product_id,
    data_hash AS stored_hash,
    SHA2(CONCAT(
      IFNULL(id, ''), '|',
      IFNULL(batch_no, ''), '|',
      IFNULL(product_id, ''), '|',
      IFNULL(DATE_FORMAT(production_date, '%Y-%m-%d'), ''), '|',
      IFNULL(remark, ''), '|',
      IFNULL(create_time, '')
    ), 256) AS current_hash
  FROM `batch`
) t
WHERE stored_hash <> current_hash;

4. 计算简化全局根哈希

提示： 项目后端根哈希是按产品 id 升序，把所有产品当前指纹用 | 拼接后再 SHA-256。 如果产品很多，可能需要调大 group_concat_max_len。
MySQL 中模拟计算 rootHash 
SET SESSION group_concat_max_len = 102400;

SELECT SHA2(GROUP_CONCAT(current_hash ORDER BY id ASC SEPARATOR '|'), 256) AS root_hash
FROM (
  SELECT
    id,
    SHA2(CONCAT(
      IFNULL(id, ''), '|',
      IFNULL(name, ''), '|',
      IFNULL(category, ''), '|',
      IFNULL(origin, ''), '|',
      CASE
        WHEN price IS NULL THEN ''
        WHEN price = 0 THEN '0'
        ELSE TRIM(TRAILING '.' FROM TRIM(TRAILING '0' FROM CAST(price AS CHAR)))
      END, '|',
      IFNULL(create_time, '')
    ), 256) AS current_hash
  FROM `product`
) t;

七、审计日志语句

审计日志表使用 previous_hash -> data_hash 模拟区块链结构。

1. 查看审计日志链

按链条顺序查看日志 
SELECT
  id,
  action_type,
  target_type,
  target_id,
  operator,
  previous_hash,
  data_hash,
  timestamp
FROM `blockchain_log`
ORDER BY id ASC;

2. 查询某个产品或批次的日志

按目标查询审计日志 
SELECT *
FROM `blockchain_log`
WHERE target_type = 'PRODUCT'
  AND target_id = 'prod_1'
ORDER BY id ASC;

SELECT *
FROM `blockchain_log`
WHERE target_type = 'BATCH'
  AND target_id = 'batch_1'
ORDER BY id ASC;

3. 查询最后一条日志

链尾日志 
SELECT *
FROM `blockchain_log`
ORDER BY id DESC
LIMIT 1;

4. 检查 previous_hash 是否连续

检查日志链上一哈希是否连续 
SELECT
  id,
  previous_hash,
  expected_previous_hash,
  CASE
    WHEN previous_hash = expected_previous_hash THEN '连续'
    ELSE '断裂'
  END AS chain_status
FROM (
  SELECT
    id,
    previous_hash,
    COALESCE(LAG(data_hash) OVER (ORDER BY id ASC), '0') AS expected_previous_hash
  FROM `blockchain_log`
) t
WHERE previous_hash <> expected_previous_hash;
说明： 项目代码验证日志链时按 id ASC 排序，而不是按 timestamp 排序。 因为日志 ID 前缀来自毫秒时间戳，和时区无关，更适合保证链条顺序稳定。

4. 模拟审计日志内容被篡改

修改某条日志 data_after 
UPDATE `blockchain_log`
SET data_after = CONCAT(IFNULL(data_after, ''), ' tampered')
WHERE id = (
  SELECT id
  FROM (
    SELECT id
    FROM `blockchain_log`
    ORDER BY id ASC
    LIMIT 1 OFFSET 1
  ) x
);

效果：审计日志验证时一般会提示“日志自身哈希不一致，可能被篡改”。

5. 模拟审计日志链断裂

删除中间一条审计日志

WHERE id = (

  FROM (
    SELECT id
    FROM `blockchain_log`
    ORDER BY id ASC
    LIMIT 1 OFFSET 1
  ) x
);

效果：下一条日志的 previous_hash 仍指向被删除日志的 data_hash， 系统会提示“上一哈希不连续，日志链已断裂”。

  FROM (

十、统计分析语句

用于展示系统首页、统计分析或答辩数据支撑。

1. 基础数量统计

产品、批次、访问、日志数量 
SELECT '产品数' AS name, COUNT(*) AS total FROM `product`
UNION ALL
SELECT '批次数', COUNT(*) FROM `batch`
UNION ALL
SELECT '溯源访问数', COUNT(*) FROM `trace_record`
UNION ALL
SELECT '审计日志数', COUNT(*) FROM `blockchain_log`;

2. 各类别产品数量

按产品类别统计 
SELECT
  category,
  COUNT(*) AS product_count
FROM `product`
GROUP BY category
ORDER BY product_count DESC;

3. 各产品批次数

产品批次数排行 
SELECT
  p.id,
  p.name,
  COUNT(b.id) AS batch_count
FROM `product` p
LEFT JOIN `batch` b ON b.product_id = p.id
GROUP BY p.id, p.name
ORDER BY batch_count DESC, p.id ASC;

4. 溯源访问排行

产品溯源查询排行 
SELECT
  p.id,
  p.name,
  COUNT(tr.id) AS trace_count
FROM `product` p
LEFT JOIN `trace_record` tr ON tr.product_id = p.id
GROUP BY p.id, p.name
ORDER BY trace_count DESC, p.id ASC;

5. 近 7 天溯源趋势

按日期统计访问数 
SELECT
  DATE(trace_time) AS trace_date,
  COUNT(*) AS trace_count
FROM `trace_record`
WHERE trace_time >= DATE_FORMAT(DATE_SUB(CURDATE(), INTERVAL 7 DAY), '%Y-%m-%d 00:00:00')
GROUP BY DATE(trace_time)
ORDER BY trace_date ASC;

6. 按操作类型统计审计日志

审计日志操作分布 
SELECT
  action_type,
  target_type,
  COUNT(*) AS total
FROM `blockchain_log`
GROUP BY action_type, target_type
ORDER BY target_type, action_type;

十一、答辩讲解口径

把 SQL 和项目功能联系起来讲，而不是只说“我会写查询”。

讲产品和批次

产品表保存基础信息，批次表通过 product_id 关联产品。 批次是溯源的关键单位，生产、质检、物流三类记录都挂在批次下。

讲溯源

溯源查询本质是多表关联：产品、批次、生产记录、质检记录和物流记录一起构成完整链路。

讲数据指纹

产品和批次都有 data_hash，通过 SQL 可以重新计算当前指纹并与存储指纹对比。

讲审计日志

blockchain_log 保存 previous_hash 和 data_hash， 通过上一哈希连续性检测是否出现日志链断裂。

推荐答辩总结： 本项目的 MySQL 不只是保存数据，还支撑了三类核心能力：第一，业务管理能力，通过产品表和批次表完成基础数据维护； 第二，溯源查询能力，通过批次关联生产、质检、物流明细；第三，防篡改验证能力，通过 data_hash 和 blockchain_log 检测业务数据和审计日志是否异常。
